## 一、统一标签体系

所有博物馆数据源通过 Adapter 层映射为统一的 `ContentItem` 数据模型，确保前端只需处理一种数据结构。

## 二、数据源配置与优先级

### Source Registry

每个数据源在 `SOURCES` 对象中注册，包含以下字段：

```javascript
const SOURCES = {
  met: {
    name: 'The Met',           // 显示名称
    flag: '🇺🇸',               // 国旗 emoji
    license: 'CC0 + 参考',     // 主要授权类型（顶部展示用；卡片维度由 norm 函数动态判定）
    color: '#e44c32',          // 品牌色（用于来源徽章背景）
    priority: 100,             // 排序权重（越高越优先）
    active: true               // 是否启用
  },
  // ...
};
```

### 当前配置

| 来源 | 定位 | 说明 |
|------|------|------|
| **The Met** 大都会 | 核心 | 全球最大开放博物馆，公共领域作品 CC0，其余仅供参考 |
| **AIC** 芝加哥 | 核心 | 公共领域作品 CC0，现当代仍在版权期作品仅供参考，API 成熟 |
| **British Museum** 大英 | 核心 | 品牌影响力最强；统一 CC-BY-NC-SA |
| **台北故宫** NPM | 核心 | 亚洲标杆，中华文化核心IP；OGDL-1.0 ≈ CC BY 4.0（署名可商用）|
| **V&A** 维多利亚与阿尔伯特 | 辅助 | 装饰艺术强项；API 无授权字段，按官方条款保守标 Non-Commercial |
| **Cleveland** 克利夫兰 | 辅助 | 6.3 万件元数据全 CC0；图片按 share_license_status 分 CC0/Copyrighted |
| **MIA** 明尼阿波利斯 | 辅助 | 无需 Key，10万+ 藏品；按 rights_type 严格判定 CC0/其余 |

### 排序算法：相关性优先交错

搜索结果不按博物馆优先级排序，而是**以搜索相关性为唯一排序依据**。

**原理**：各家 API 返回的结果内部已经按相关性排好序（第 1 条最相关）。合并时按"排名"交错取值：

```
第 1 轮：Met 的第1名, AIC 的第1名, BM 的第1名, NPM 的第1名, V&A 的第1名, CMA 的第1名
第 2 轮：Met 的第2名, AIC 的第2名, BM 的第2名, ...
第 3 轮：Met 的第3名, ...
...
```

- 某一源在某轮没有更多结果时自动跳过，不补位
- 不引入任何权重或优先级偏向
- 效果：各源最相关的结果集中出现在前几排，不相关的自然沉底

### 如何新增数据源

1. 在 `SOURCES` 中添加注册信息（含 priority）
2. 编写 `normXxx(o)` 归一化函数，输出 `ContentItem` 格式
3. 编写 `xxxAdapterSearch(q, maxItems)` 搜索函数
4. 在 `multiSourceSearch()` 中添加 adapter 调用
5. 在 Header 的 `#sourceTabs` 中添加对应 Tab 按钮
6. 在侧栏筛选器中添加对应 checkbox

## 三、授权标签体系

### 设计原则：外显三态，内部细分

**外显层（卡片徽章）三态分类**——「先看版权，再看商用」：

| 状态 | 徽章 | 含义 | 灵感参考 | 商用 |
|:---:|:---:|:---|:---:|:---|
| 🟢 `com` (可商用) | 绿色 | 作品已进入公共领域 或 来源馆采用开放许可 | ✅ | ✅ 直接用 |
| 🟡 `copyright` (受版权保护) | 琥珀黄 | 作品本身仍在版权保护期内（如现当代艺术品）| ✅ | ❌ 需联系**版权方**申请授权 |
| ⚪ `restricted` (来源馆限制) | 灰色 | 作品多为公共领域，但**馆方**对图像主张权利或状态未明 | ✅ | ⚠️ 可联系**来源馆**付费授权 |

**为什么分三态**：黄色 vs 灰色虽然都是"现在不能直接商用"，但**未来路径不同**——黄色需要等版权过期/联系版权方（往往是死路），灰色可以走馆方付费授权（有路）。这个差异对决策非常重要。

**核心认知**：浏览/做灵感参考时**三类全可用**；只有要商用时才需要按颜色区分路径。

### 颜色配色

| 状态 | 颜色码 | CSS class（徽章） | CSS class（详情卡片） |
|:---:|:---:|:---:|:---:|
| `com` | 🟢 `#008a05` | `badge-license-com` | `license-card.com` |
| `copyright` | 🟡 `#d97706` | `badge-license-copyright` | `license-card.copyright` |
| `restricted` | ⚪ `#9aa0a6` | `badge-license-restricted` | `license-card.restricted` |

### 三态判定规则（`licenseStatusOf`）

```js
function licenseStatusOf(license, source) {
  if (license.com === true) return 'com';                                     // 1) 可商用
  if (['Reference', 'Copyright', 'In Copyright'].includes(license.type))
    return 'copyright';                                                        // 2) 作品本身受版权保护
  return 'restricted';                                                         // 3) 来源馆限制（默认）
}
```

**内部细分（`license.type` 字段）保留 9 种细粒度值**，用于：
- 详情弹窗版权信息卡片的精确说明（严格按各馆官方条款 / Creative Commons 原文）
- 卡片徽章 `title` 属性 hover 提示
- 后续可能的高级筛选

### 详情弹窗：「使用建议」横幅 + 官方条款细则

详情弹窗的「版权信息」区分两层展示：

**第一层（使用建议横幅，按 status 分类）**：
- 🟢 可商用：作品已进入公共领域或采用开放许可，可放心用于任何用途，包括商业项目。
- 🟡 受版权保护：作品本身仍在版权保护期内。可作为内部灵感参考与浏览，但商业使用需联系版权方申请授权。
- ⚪ 来源馆限制：作品本身可能为公共领域，但来源馆对其图像主张机构权利或版权状态未明。可作为内部灵感参考与浏览，商业使用建议联系来源馆申请付费授权。

**第二层（细则卡片，按 license.type 分类）**：
按 `licNoteMap` 给出 CC0 / CC BY 4.0 / Reference / Non-Commercial / CC-BY-NC-SA / Copyright / Public Domain (Not Evaluated) / Rights Not Evaluated 等各自的官方条款摘要 + 链接。

### V&A 调研记录（为何统一标 Non-Commercial）

**API 实测**：调用 `api.vam.ac.uk/v2/objects/search` 返回的记录字段仅含：
```
systemNumber, accessionNumber, objectType, _primaryTitle, _primaryMaker,
_primaryImageId, _primaryDate, _primaryPlace, _images
```
**完全不包含任何 license / rights / copyright 字段**，无法逐件判定授权状态。

**官方条款（Section 9）摘录**：
> "If you wish to use Content for any other purpose (i.e. for a use that is commercial or for a use not covered by the Explore the Collections terms and conditions), please contact our Image Licensing"

V&A 商业图像授权走 [vandaimages.com](https://www.vandaimages.com/) 付费渠道。因此整体保守标注为 `Non-Commercial`。

### NPM 授权依据

台北故宫 OpenData 整体采用 **OGDL-1.0（政府资料开放授权第 1 版）**，规则等价于 **CC BY 4.0**：
- 可商用、可衍生、可再分发
- 必须保留来源说明（推荐："来源 国立故宫博物院"）
- 不得歪曲、删除原始版权声明

虽然 NPM 官方对个人下载声明"无需申请、不限用途、不需付费"，**保守按 CC BY 4.0 标注**（com=true 但需署名），更符合"严格者优先"原则。

### UI 展示规则

- **卡片右上角徽章**：按三态显示——`com` 绿、`copyright` 黄、`restricted` 灰；文案为 `licenseBadgeLabel(type)`（如 "CC0 可商用" / "署名可商用" / "受版权保护" / "非商用" / "未评估的公共领域"）
- **徽章 hover tooltip**：显示原始英文 `license.type`（如 `CC BY 4.0`、`Reference`）
- **详情弹窗版权区**：两层结构 — 第一层「使用建议横幅」按 status 三态用通俗中文说明灵感参考与商用边界；第二层「细则卡片」按 license.type 给出官方条款摘要 + Creative Commons / 来源馆条款官方链接
- **数据源顶部 `SOURCES.license` 字段**：表达该源整体特征（如 `'CC0 + 版权混合'` 说明含 PD 与版权两类，`'CC BY 4.0'` 表示统一署名可商用，`'非商用'` 表示统一馆方限制）

### 设计决策：保留非商用作品作为灵感参考

非 CC0/CC BY 作品（Met/AIC 现当代藏品、Cleveland Copyrighted、V&A 全部、MIA Copyright 与 Not Evaluated 系列、BM 全部）**仍接入展示**。理由：
- **灵感参考层面三类全可用**：浏览、内部 mood board、学习研究都在合理使用范围
- **决策时刻才需区分**：用户决定要拿这张图做商用时，按三态颜色判断不同路径——绿色直接用、黄色等版权过期/找版权方、灰色找馆方付费授权
- **UI 双层提示**：徽章三态视觉区分 + 详情弹窗使用建议横幅强化"灵感参考全可用，但要谨慎商用"的认知

## 三·A、文化主题与资产形式双维度标签体系

> 本章为 2026-05-13 引入的产品架构升级：从「博物馆资产库」扩展为「文化资产库」，引入 `asset_type`（资产形式）+ `themes`（文化主题）双维度标签体系，为未来接入 3D 模型、文本、音视频等非博物馆类资产预留结构。
>
> 完整产品定位与架构讨论见 `../文化资产库_产品信息架构设计.md`。本章只讲实现规则。

### 1. 字段定义

在 `ContentItem` 统一数据模型中新增两个字段：

| 字段 | 类型 | 说明 | 默认值 |
|---|---|---|---|
| `asset_type` | string（单选） | 资产形式：`image` / `3d_model` / `text` / `audio_video` / `dataset` | `image`（现阶段全部博物馆数据） |
| `themes` | string[]（多选） | 文化主题，可挂多个 | `['others']`（兜底，无任何特殊命中时） |

### 2. themes 标签清单（9 个，终版）

每个 theme 有**明确的语义边界**，不是纯分类，而是"创作者检索时会想要的调性索引"。

| 标签值 | 中文名 | 语义边界 | 典型命中对象 |
|---|---|---|---|
| `landmark` | 地标建筑 | 具有标志性的建筑/遗址/石窟/宫殿/宝塔 | 莫高窟、故宫、金字塔、罗马斗兽场 |
| `landscape` | 风景地貌 | 自然风景、山水画、地图、地理影像 | 宋代山水画、富岳三十六景、古地图 |
| `mythology_religion` | 神话与信仰 | 跨文化的神祇/宗教/神话/圣像/经典母题 | 佛像、菩萨、十字架、希腊众神、山海经神兽 |
| `historical_figure` | 历史/文学/神话人物 | 可识别的具名人物（非泛指） | 孔子、关羽、菩萨、基督、法老图坦卡蒙 |
| `folk_custom` | 民俗节庆 | 节庆活动、民俗仪式、婚俗、傩戏 | 春节、龙舟、庙会、傩面具 |
| `regional_culture` | 文化圈/文化人格 | **有专属审美体系与符号系统的文化簇**（非国别） | 维京、武士、丝路胡风、藏传、楚文化、凯尔特 |
| `craft_heritage` | 工艺与非遗 | 以工艺为核心的文物 + 活态非遗记录（两端合并） | 漆器、景泰蓝、花丝、缂丝、刺绣、木雕 |
| `expert_resource` | 专家与口述 | 学者访谈、口述史、研究论文 | 非遗传承人访谈、考古学报告 |
| `others` | 其他 | **兜底**：未命中任何上述 8 个 theme 的资产 | 未特化的博物馆藏品（用于发现词典缺口） |

> ⚠️ **`regional_culture` 的关键语义**：它**不是地理国别**（"中国/日本"不触发），而是"具有独特文化人格的文化圈"——判断标准是"有专属审美、专属符号、专属群体认同"其中一条。维京、武士、丝路、楚巫都符合；单纯"中国陶瓷"不符合。

> ⚠️ **方案 C：移除 `museum_collection` 作为筛选 theme**（2026-05-13 修订）
>
> 原方案保留 `museum_collection` 作为博物馆数据兜底，所有博物馆数据默认挂这个 theme。但 UI 测试发现：
> 1. 它的数字永远等于"总数"，对用户而言是冗余信息
> 2. 它的语义跟其他 8 个"题材调性"不在一个维度（"来自博物馆"是数据来源属性，不是题材属性）
> 3. 它的存在让"文化主题加总 > 总数" 的偏差更大（每件至少多挂 1）
>
> 修订为：**移除 `museum_collection` 作为筛选 theme，引入 `others` 作为兜底**。
> - 未命中任何 8 个特殊 theme 的资产 → 自动挂 `others`
> - 每件资产至少挂一个 theme（保持加总更接近总数）
> - 各 theme 命中数加总仍可能略大于总数（因为部分资产挂多个 theme，这是多选标签的必然特性），但 `others` 让"完全未命中"数量显式可见，对发现词典缺口仍然有效
> - `museum_collection` 作为概念保留在产品架构文档（用于未来与活态非遗/当代创作做资产性质对偶），但**不进 themes 数组、不进筛选 UI**

### 3. 打标方法论：词典驱动 + 兜底 + 观测

**核心原则：开放集合，不硬编码**。打标规则必须满足三条：

1. **未命中不报错**：任何资产都能被打标，没命中任何特殊 theme 的归 `others` 兜底
2. **词典可延展**：规则集中在一份外置词典里，增加一个新文化圈只需补关键词，不改代码
3. **命中率可观测**：UI 上实时显示当前结果的 theme 分布，让"词典漏洞"主动浮现

#### 3.1 词典形态选型

选项对比：

| 形态 | 优点 | 缺点 | 何时切换 |
|---|---|---|---|
| **内嵌 JS 对象**（当前选择）`const THEME_DICT = {...}` | 零依赖、随代码版本控制、一眼看懂 | 文件膨胀；词典修改要动 html | 词典 ≤ 几百词 |
| 独立 JSON 文件 `themes-dict.json` + fetch | 数据与代码彻底分离、可交运营维护 | 需异步加载、首屏延迟、需额外工具校验 | 词典膨胀到上千词 或 交由非开发者维护 |
| 服务端词典表 | 可热更新、可按用户定制 | 需要后端支持、脱离"纯前端 demo"形态 | 产品正式立项后 |

**当前选择：内嵌 JS 对象**。  
**理由**：现阶段是纯前端 demo，词典规模预估 ~100 关键词，嵌入代码的运维成本最低；词典未来膨胀后可通过简单的 `module export` 拆到独立文件，当前设计预留了这个扩展点。

#### 3.2 词典数据结构


**关键设计点**：
- **分语言子键**：`keywords.en` + `keywords.zh` 允许未来扩展中文数据源（非遗词库、中文古籍）不污染英文词表
- **`gaps` 字段**：显式承认"当前词典不覆盖什么"，是可延展性的文档化表达。每次 review 时优先看 `gaps` 决定是否补充
- **`label` 字段**：筛选 UI 直接用这个展示中文名，词典 = 唯一真相源

#### 3.3 打标器（代码稳定，不随词典增改而动）

```js
function enrichThemes(item) {
  const themes = [];
  const haystack = [
    item.classification, item.title, item.desc || item.description,
    item.culture, item.period, item.medium
  ].filter(Boolean).join(' ').toLowerCase();
  
  for (const [theme, def] of Object.entries(THEME_DICT)) {
    const allKeywords = [...(def.keywords.en || []), ...(def.keywords.zh || [])];
    if (allKeywords.some(kw => haystack.includes(kw.toLowerCase()))) {
      themes.push(theme);
    }
  }
  // 兜底：没有命中任何特殊 theme → 挂 others
  if (themes.length === 0) themes.push('others');
  return [...new Set(themes)]; // 去重
}
```

这个函数**纯粹遍历词典**，不含任何硬编码业务规则。词典增删改完全不需要改它。

#### 3.4 未命中观测（UI 层可延展性保障）

**问题**：如果词典覆盖不全，数据静悄悄归入兜底，维护者永远不知道漏了什么。

**解决**：把"筛选项命中数"设计为一等公民——**左侧筛选栏的每一个 checkbox 旁都显示当前命中数**，把数据分布透明化。

```
文化主题  (一件可挂多个)
├─ ☐ 地标建筑 (15)
├─ ☐ 风景地貌 (22)
├─ ☐ 神话与信仰 (8)
├─ ☐ 历史/文学人物 (3)
├─ ☐ 民俗节庆 (0)          ← 信号：可能需要补词典
├─ ☐ 文化圈 (12)
├─ ☐ 工艺与非遗 (47)
├─ ☐ 专家与口述 (0)
└─ ☐ 其他 (未特化) (113)    ← 兜底：词典未命中的数量，发现漏词的关键指标
```

**命中数解读**：
- 所有 themes 加总 ≥ 总数（因为部分资产挂多个 theme——例如敦煌飞天同时是 landmark + mythology + regional_culture）
- `其他 (未特化)` 数字越高 → 词典覆盖率越低 → 有补词机会
- 某个特化 theme 长期为 0 → 可能词典漏词 或 当前筛选组合下确实无供给

**命中数计算规则（B 方案，联动筛选）**：
- 每个选项旁的数字 = 在"当前其他维度筛选都已应用"的前提下，单独勾选自己能返回的条数
- 维度间正交：本维度内部勾选不影响自己的数字
- 用户体验理由详见 `../文化资产库_产品信息架构设计.md` 第三·5 章

**代码实现**：
```js
function updateFilterCounts() {
  const all = state.allSourceItems || [];
  if (!all.length) return;
  // 对每个维度，排除自身做一次 filterItems，再对本维度每个选项计数
  const srcPool   = filterItems(all, 'source');
  const periodPool = filterItems(all, 'period');
  const geoPool   = filterItems(all, 'geo');
  const licPool   = filterItems(all, 'license');
  const themePool = filterItems(all, 'theme');
  // ... 分别计数并写回 UI
}
```

其中 `filterItems(items, skipDim)` 的 `skipDim` 参数是关键：计算某维度命中数时暂时跳过"自身维度"的过滤，得到"其他维度已筛过"的候选池，再对本维度每个选项做单项匹配。

**应用范围**：此机制**同时覆盖 5 个筛选维度**——数据来源、时期、文化地域、授权三态、文化主题。每次搜索结果回来或任一 checkbox 变化都重新计算。

**数值意义**：当某个 theme 的命中数**长期为 0** 或**与预期不符**时，就是词典该扩充的信号。

### 4. 新主题决策原则

**优先扩词典，慎开新 theme**。

当一批新素材（比如接入三星堆数据）难以归入现有 theme 时，按以下顺序判断：

1. **语义相近？** → 扩词典。三星堆可归入 `regional_culture`（新增"古蜀文化圈"子组） + `mythology_religion`（青铜神树）
2. **完全正交且量级可观？** → 才考虑开新 theme。例如未来接入"当代二创"类，才开 `contemporary_creation`
3. **每新开一个 theme 必须更新**：（a）THEME_DICT；（b）本章标签清单；（c）筛选 UI；（d）产品架构文档

### 5. 现阶段打标的默认值约定

所有博物馆 API 数据：
- `asset_type = 'image'`（目前全部是图片）
- `themes` 初始为空，由 `enrichThemes()` 根据词典追加；任何未命中的资产挂 `['others']`
- 一件资产可能同时命中多个特殊 theme（如 landmark + mythology_religion）——这是**特性**，支持多维度检索

未来新数据源要覆盖：
- Sketchfab 3D 模型 → `asset_type = '3d_model'`，`themes` 根据元数据打标
- 古籍扫描 → `asset_type = 'text'`
- 非遗视频 → `asset_type = 'audio_video'`，默认追加 `craft_heritage` 或 `intangible_heritage` 相关 theme

## 四·A、时期匹配规则（区间重叠 + 文本兜底）

> 2026-05-13 升级。原"时期"筛选靠关键词字符串匹配（"medieval"、"tang"），但博物馆 API 返回的 `dateDisplay` 字段绝大多数是纯年份（"1475-1490"、"c. 1937"），两者对不上 → 大量藏品漏匹配。B 方案命中数开启后问题立刻暴露（如 270 件结果中只有 ~61 件被时期识别）。

### 设计：每个时期 checkbox 配年份区间

HTML 里每个 `[data-filter="period"]` 的 `value` 从"关键词字符串"改为 `"起始年,结束年"`（BCE 用负号），前端按**区间重叠**命中。

**世界通用时期（西方/全球史框架）**：

| 时期 | 区间 | 依据 |
|---|---|---|
| 新石器时代 | `-10000,-3000` | 含中东/东亚早期新石器；实际藏品密集在 -3500~-2000 |
| 上古 | `-3000,-500` | 埃及王朝 -3100、苏美尔 -3000、商周、希腊古风期 |
| 古典 | `-500,500` | 希腊古典、罗马帝国、汉代跨两段（区间重叠是正确现象）|
| 中世纪 | `500,1400` | 罗马陷落 476 → 文艺复兴起点 1400 |
| 文艺复兴 | `1400,1600` | 意大利早期文艺复兴 ~1400；北方延至 1600 |
| 近世 | `1600,1800` | 巴洛克、启蒙时代 |
| 近现代 | `1800,2100` | 19 世纪起至今 |

**中国朝代**：

| 朝代 | 区间 | 备注 |
|---|---|---|
| 商周 | `-1600,-256` | 商 -1600~-1046 + 西周 -1046~-771 + 东周 -770~-256 |
| 汉代 | `-206,220` | 西汉 + 新莽 + 东汉（按汉系统延续处理） |
| 唐代 | `618,907` | 标准断代 |
| 宋代 | `960,1279` | 北宋 + 南宋 |
| 元代 | `1271,1368` | **保守取 1271**（忽必烈定国号"元"）；前蒙古帝国不算 |
| 明代 | `1368,1644` | 标准断代 |
| 清代 | `1644,1912` | 入关 ~ 辛亥革命 |

**日本时代**：江户 `1603,1868` / 明治 `1868,1912`

### 匹配三级回退

`matchPeriod(item, periodValue)` 按以下顺序命中：

**① 优先：`dateRange` 区间重叠**  
`item.dateRange.begin/end` 与 checkbox 区间有交集即命中。Met / AIC 的 norm 函数都提供此字段。

```js
return itemEnd >= periodStart && itemBegin <= periodEnd;
```

**② 回退：从 `dateDisplay` 抽年份**  
用 `extractYearFromText()` 提取首个 3-4 位数字，识别 BCE / BC / 公元前 / 前 等 BCE 标记转负号。抽出年份落入区间即命中。

适用场景：V&A、BM、MIA 等 `dateRange` 缺失但 `dateDisplay = "1475-1490"` 的藏品。

**③ 文本兜底：学术语词匹配**  
`PERIOD_TEXT_KEYWORDS` 给每个区间定义一组学术语词（如 `"500,1400": ['medieval','gothic','byzantine','tang dynasty','song dynasty']`），在 `culture / period / dynasty / title` 拼接文本中做 `includes` 匹配。

适用场景：既无 `dateRange` 又无数字年份，但 `culture = "Tang Dynasty"` 这类学术词命名的藏品。

### 区间重叠是特性，不是 bug

一件 1450 年的挂毯会**同时**命中"中世纪 (500-1400)" 和 "文艺复兴 (1400-1600)"——这正是历史断代的真实样貌（过渡期文物本来就跨两段）。因此：
- "找到 X 件藏品" ≠ 各时期命中数之和
- **时期维度的加总 ≥ 总数** 是正常的

### API 查询补强

`performSearch` 构造 API 查询串时，不再把区间字符串直接拼入（那会乱），而是用 `PERIOD_TEXT_KEYWORDS` 反查纯英文学术词作为查询补强：

```js
const periodKeywords = (state.filters.periods || [])
  .flatMap(p => PERIOD_TEXT_KEYWORDS[p] || [])
  .filter(k => /^[a-z\s\-]+$/i.test(k))   // 只保留英文，过滤中文兜底词
  .join(' ');
```

这样用户勾"宋代"，API 查询里会加 `"song dynasty northern song southern song 宋"`（过滤后只剩英文部分），既帮助 API 相关性排序，又不污染客户端的区间判定。

### 已知局限

- **"文化地域"没有类似机制**：藏品 `culture` / `geo.country` 字段写法多样（"German"、"Tournai"、"Gothic (medieval)"），跟固定 8 国列表匹配度低。此字段暂不做区间化改造（不适用），建议后续按需扩 culture → country 词典。
- **日本飞鸟/奈良/平安等时代缺失**：当前 UI 只列江户和明治，其他日本时代藏品进不了中国/欧洲区间，会落空。待 UI 扩展时再补。

## 五、搜索相关性优化

### 中文搜索自动翻译（必选项）

> **设计规则：所有搜索入口必须经过中英文翻译处理。** 这是硬性要求，不可省略。

所有外部 API 的元数据都是英文，中文搜索需自动翻译为英文。采用两层机制：

1. **本地映射表（即时）**：覆盖 80+ 常用中文词（分类/朝代/文化/材质/主题）
2. **Google Translate 兜底（异步）**：映射表未覆盖的词调用 `translate.googleapis.com` 免费端点

```
用户输入 "中国瓷器"
  → 本地映射：中国→"Chinese China"，瓷器→"ceramics porcelain"
  → 合并为："Chinese China ceramics porcelain"
  → API 源用英文查询，本地源（大英/台北故宫）用中英文同时匹配
```

用户输入任意中文（如 "波斯地毯"、"明代青花"）都能自动翻译后搜索，不局限于映射表。

### Met API 客户端过滤

Met API 的搜索是全文模糊匹配（如搜 "Chinese ceramics" 会返回任何含 "Chinese" 的结果）。为提升精准度，Met Adapter 增加了**客户端相关性过滤**：

```
搜索词 → 拆分为关键词数组（忽略 ≤2 字符的词）
API 返回结果 → 检查 title/culture/classification/medium/period/department 是否包含至少一个关键词
不匹配的结果 → 过滤掉
```

为补偿过滤损耗，API 请求量为目标数的 3 倍（`maxItems * 3`），确保最终结果数充足。

### 预构建数据搜索

大英博物馆和台北故宫的本地数据采用客户端关键词匹配：
- 搜索词拆分为关键词
- 在 `title + culture + classification + medium + country + period + desc` 中匹配
- 任一关键词命中即视为匹配

### 搜索结果数量上限

**每家数据源每次搜索最多返回 100 条结果**，这是综合考虑后的最佳值。

| 来源 | 上限 | 限制原因 |
|------|:----:|---------|
| The Met | 100 | API 总量充足（数万），瓶颈在每条需单独 fetch 详情，100 条约需 5-10 秒 |
| AIC | 100 | API 单次请求硬上限 100 |
| Cleveland | 100 | API 单次请求硬上限 100 |
| V&A | 100 | API 单次请求硬上限 100 |
| 大英博物馆 | 100 | 预构建数据总量约 22 条，实际取全部 |
| 台北故宫 | 100 | 本地数据集 340 条，按相关性匹配 |

**为什么不全量返回**：
- Met 全量约 2 万+ ID，每条单独请求需几十分钟，浏览器会卡死
- 前端一次渲染数千张卡片会严重卡顿
- AIC/CMA/V&A 的 API 硬性限制单次请求 ≤100 条

**实际效果**：6 源最多并行返回 600 条，足以支撑前端筛选和分页展示。如需更多，后续可加分页加载（infinite scroll）。

| 台北故宫 | 本地 `assets/npm/<id>.jpg` | 同图 | 同图 |

### Fallback 规则

1. 优先使用 `thumb`（缩略图），加载快
2. `thumb` 为空时 fallback 到 `full`（高清图）
3. 图片加载失败时显示 🖼️ 占位符（`onerror` 处理）
4. 只保留有图片的结果（无图 = 不展示）
