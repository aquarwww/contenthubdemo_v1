## 2026-05-12｜台北故宫接入本地数据集（解决图片拦截）

### 背景
原 NPM Adapter 使用 8 件 Wikimedia 远程图（`upload.wikimedia.org`），被公司网络拦截，卡片只能显示占位符。本地已下载「故宫资料3万余件台北故宫藏品高清图片素材文字说明」（约 23k 条 GB18030 编码 txt + 20k jpg），可直接利用。

### 改动

**1. 新增数据抽取脚本 `build-npm-local.js`**
- 扫描 `[器物典藏](32072件)` 下 11 个子类目（青铜/玉石/漆器/珐琅/竹木牙角/书画/文房/杂项等）
- **智能编码识别 `readSmartText`**：数据集是混编的（早期 GB18030、后期 UTF-8），先尝试 UTF-8，若解不出"品名"关键词则回落 GB18030
- 解析「品名/類別/質材/功能/尺寸/時代/說明」字段，第二行夹注（西元年份/英文标题）一并提取
- 配对同前缀的 jpg（优先 PAA/PAB 主图），各子类目按配额均匀采样
- 配对图片复制到 `assets/npm/<id>.jpg`（ASCII 文件名 + 中文前缀字母代号防撞 ID）
- 输出 `npm-local.json`：**340 条**精选条目，含中英文标题、时代（中文朝代+西元年份说明）、详细说明

**2. 改写 `content-hub-demo.html` 的 NPM Adapter**
- 移除硬编码的 `NPM_FEATURED`（8 件 Wikimedia 远程图）
- 改为 `loadNPMLocal()` 异步 fetch `npm-local.json`，挂在 `NPM_LOCAL`
- `normNPM` 适配新字段：
  - `title = titleEn ? '中 · En' : titleZh`（无英文时直接展示中文/繁体）
  - `dateDisplay = '中文朝代 · 西元年份'`（如"商 · 西元前16-前11世紀"）
  - desc 拼接 质材/尺寸/说明
- `npmAdapterSearch` 同时匹配 中英文标题、时代、类别、说明、tagZh

### 效果
| 维度 | 改造前 | 改造后 |
|------|--------|--------|
| 数据量 | 8 件 | **340 件** |
| 图片可访问性 | ❌ wikimedia 被拦截 | ✅ 本地 assets/npm/ |
| 中文搜索 | 命中率低（仅 8 件） | 大幅提升（340 件，11 类目） |

| 时代信息 | 8 条粗略 | 301 条含西元年份精确说明 |

### 数据质量说明
- 抽样按子类目均匀分布，11 个类目按配额抽取 340 条
- 数据集**编码混编**：早期 txt 是 GB18030，后期 txt 是 UTF-8 → 用 `readSmartText()` 自动检测（先尝试 UTF-8 解出"品名"关键词，否则回落 GB18030）
- `dateEn` 规则：保留所有非纯数字的第二行（"西元前11世紀中葉-前10世紀中葉"、"西元1701-1800年" 等真实年份说明都保留；"1943" 等孤立数字判定为入藏年/编号噪音，丢弃）
- `titleEn` 缺失时直接展示中文（含繁体）—— `normNPM` 中 `title = titleEn ? '中 · En' : zh`
- ID 唯一性：`类目英文-accession中文前缀字母代号-数字编号`（如 `npm:jade-np-000014-000000000`），并加 dupCnt 兜底

## 2026-05-12｜许可证标注精确化（CC0 + 仅供参考 双轨）

### 背景
此前对 The Met / AIC / Cleveland 三家美国馆笼统标注为 CC0，但**实际它们都是"部分 CC0"**：
- Met / AIC：单件作品的 `isPublicDomain` / `is_public_domain` 字段决定（true → CC0，false → 仍在版权期，如毕加索、马蒂斯等现当代）
- Cleveland：`share_license_status` 字段（"CC0" / "Copyrighted"）

旧实现里，AIC/CMA Adapter 在 API 查询层强制 `is_public_domain=true` / `cc0=1` 把非 CC0 作品全过滤掉，因此结果集都是 CC0、标签也准确。但**这等于丢掉了大量优质参考素材**（Met 现当代藏品、CMA 的 Copyrighted 一半藏品等）。

### 决策
**保留非 CC0 作品作为"仅供参考"接入**，与现有 BM (CC-BY-NC-SA)、V&A (Mixed) 的处理方式对齐：放进搜索结果，但用清晰的视觉与文案区分商用资格。

### 改动
**1. 三态 license 体系（新增 Reference）**
- `licenseBadgeClass` / `licenseBadgeLabel` 增加 `Reference → 仅供参考`（紫色 `#8e7cc3`）
- 新增辅助函数 `licenseBadgeFullClass(type)` 做卡片徽章 CSS 类映射
- CSS 新增 `.badge-license-reference` 与 `.license-card.reference` 两组样式

**2. norm 函数动态判定**
- `normMet`：`license.type = isPublicDomain ? 'CC0' : 'Reference'`（原值 `'Restricted'`）
- `normAIC`：同上，按 `is_public_domain`
- `normCMA`：改读 `share_license_status === 'CC0'`（原值硬编码为 CC0）

**3. Adapter 不再强制过滤**
- AIC：去掉 `query[term][is_public_domain]=true` 参数
- CMA：去掉 `cc0=1` 参数
- Met：本来就没强制过滤（只在用户勾选"仅公共领域"时才过滤），保持现状

**4. 卡片徽章文案优化**
- 显示用户友好的中文标签（"CC0 可商用" / "仅供参考"），原始 type 放在 `title` 属性供 hover

**5. 详情弹窗版权卡片**
- 四态配色（CC0 绿 / CC BY 橙 / Reference 紫 / Restricted/CC-BY-NC-SA 红）
- 每种类型对应清晰的中文合规说明文案（如 Reference → "⚠️ 仍在版权期内 · 仅供内部学习参考，不可商业使用、不可下载复用"）

**6. SOURCES 注册表**
- Met / AIC / CMA 的 `license` 字段从 `'CC0'` 改为 `'CC0 + 参考'`，明示混合授权

## 2026-05-13｜授权体系简化为两态 + 各馆精细化

### 背景
上一版（2026-05-12）做了"CC0 + Reference"双态接入。本轮针对几个具体不准确点继续精细化：
1. **配色简化**：紫色 Reference + 红色 Restricted 并存视觉混乱 → 统一灰色，由 `license.com` 决定颜色
2. **MIA 处理**：`Public Domain (Not Evaluated)` 之前被算作 CC0（不严谨），且非 PD 作品没标 Reference
3. **NPM 标注**：之前是 `CC0/CC BY 4.0` 含糊，未取严格者
4. **V&A 标注**：硬编码 CC0 完全错误，需查证官方条款后修正

### V&A 调研（结论：API 无授权字段，整体 Non-Commercial）

**API 实测**：调用 `api.vam.ac.uk/v2/objects/search` 返回的记录字段仅含：
```
systemNumber, accessionNumber, objectType, _primaryTitle, _primaryMaker,
_primaryImageId, _primaryDate, _primaryPlace, _images
```
**没有任何 license / rights / copyright 字段**，无法逐件判定。

**官方条款**（[vam.ac.uk Section 9](https://www.vam.ac.uk/info/va-websites-terms-conditions)）：
> "If you wish to use Content for any other purpose (i.e. for a use that is commercial...), please contact our Image Licensing"

商业图像授权走 [vandaimages.com](https://www.vandaimages.com/) 付费渠道。**保守统一标 `Non-Commercial`**。

### NPM 授权依据修正

台北故宫 OpenData 整体采用 **OGDL-1.0**（政府资料开放授权第 1 版），规则等价于 **CC BY 4.0**：可商用、可衍生，但需署名。

### 可延展性机制（v2 核心创新）

## 2026-05-13（下午 5）｜双维度标签体系代码落地 + 筛选项命中数（B 方案）

### 新增设计原则：筛选项即数据分布图

起源于 PM 观察到"找到 218 件藏品（AIC: 100 · Cleveland: 7 · V&A: 25 · MIA: 86）"这个已有的顶部结果计数行——本质上是一种"数据分布可视化"。推演后：这种能力应当**推广到左侧所有筛选项**，让每个 checkbox 旁都带一个当前命中数。

**命中数语义：B 方案（联动筛选）**
- 每个选项旁数字 = 在其他维度筛选都应用的前提下，单独勾选本选项能返回的条数
- 维度之间正交，本维度内部勾选不影响自己数字
- 采用这种联动而非完全独立：当用户缩小到唐代结果后，看到的 theme 命中数应反映这个子集

顶部结果计数行相应简化：
- 保留：`找到 218 件藏品`（一目了然的完成反馈）
- 删除：`（AIC Chicago: 100 · Cleveland: 7 · V&A Museum: 25 · MIA Minneapolis: 86）`（信息下沉到左侧"数据来源"筛选项的命中数里）

## 2026-05-13（下午 6）｜时期匹配升级为区间重叠（修复命中缺口）

### 背景：B 方案命中数暴露的漏匹配问题

上一版时期筛选是靠英文关键词字符串匹配（`"medieval"`、`"tang"`）。开启 B 方案命中数后立刻暴露严重漏匹配：搜 "medieval European tapestry" 返回 270 件，但左侧"中世纪 52 + 文艺复兴 1 + 近现代 8 = 61"——**绝大多数藏品的时代信息压根对不上 checkbox**。

根本原因：博物馆 API 返回的 `dateDisplay` 绝大多数是**纯年份数字**（`"1475-1490"`、`"c. 1937"`、`"before 1489"`），但我们的匹配关键词是**时代名称**（"medieval"、"renaissance"）。两者无法 `includes` 命中。

旧版没暴露是因为：
- 旧版没有命中数显示
- 旧逻辑把时期关键词拼入搜索 query 发给 API，靠 API 全文搜索兜底命中

新版改成客户端过滤 + B 方案命中数，漏洞直接可见。

### 设计：区间重叠 + 三级回退

每个时期 checkbox 的 `value` 从 `"Medieval"` 改为 `"500,1400"`（区间字符串，BCE 用负号）。

**断代精确度（学术主流）**：
- 世界通用：新石器 `-10000,-3000` / 上古 `-3000,-500` / 古典 `-500,500` / 中世纪 `500,1400` / 文艺复兴 `1400,1600` / 近世 `1600,1800` / 近现代 `1800,2100`
- 中国朝代：商周 `-1600,-256` / 汉 `-206,220` / 唐 `618,907` / 宋 `960,1279` / 元 `1271,1368` / 明 `1368,1644` / 清 `1644,1912`
- 日本：江户 `1603,1868` / 明治 `1868,1912`
- **元代保守取 1271**（忽必烈定国号），前蒙古帝国不算
- **接受区间重叠**：1450 年挂毯同时命中中世纪 + 文艺复兴，这是历史断代的真实样貌

**匹配三级回退**（新函数 `matchPeriod`）：

| 级 | 依据 | 适用 |
|---|---|---|
| ① | `item.dateRange.begin/end` 区间重叠 | Met/AIC 有该字段 |
| ② | `extractYearFromText(dateDisplay)` 抽首个 3-4 位数，识别 BCE/BC/公元前/前 转负号 | V&A/BM/MIA 的 `"1475-1490"` 类 |
| ③ | `PERIOD_TEXT_KEYWORDS` 文本兜底 | 有 `culture="Tang Dynasty"` 但年份缺失的藏品 |

**文本兜底关键词词典** `PERIOD_TEXT_KEYWORDS` 覆盖 16 个区间，约 80 个学术语词（medieval/gothic/tang dynasty/宋/edo period 等），同时含中文。

### 代码改动

1. **HTML**：17 个时期 checkbox 的 `value` 全部替换为区间字符串
2. **新增 `PERIOD_TEXT_KEYWORDS` 词典**（紧接 THEME_DICT 之后）
3. **新增 `extractYearFromText(s)`**：识别 BCE 标记 + 抽取年份数字
4. **新增 `matchPeriod(item, periodValue)`**：三级回退匹配
5. **`filterItems` 里 period 分支**：改用 `matchPeriod`
6. **`updateFilterCounts` 里 period 分支**：改用 `matchPeriod`
7. **`performSearch` 里 API 查询构造**：不再直接拼区间字符串；改为反查 `PERIOD_TEXT_KEYWORDS` 取英文学术词（过滤掉中文兜底词）拼入 query，帮助 API 相关性排序而不污染客户端判定

### 附带改动

**文化主题标题旁加"一件可挂多个"小字提示**（`.filter-hint` 新样式）：  
避免用户看到"landmark + mythology + regional 加总 > 总数"时误以为是 bug。多选标签的加总必然 ≥ 总数，这是设计特性而非缺陷。

- 如果 "其他" 数字接近总数（比如 380/400），说明词典覆盖率低，该补词了
- 如果 "其他" 数字小（比如 50/400），说明词典命中率健康

### 已知现象（设计如此）
- 8 个特殊 theme 加总 + 其他 仍可能 > 总数（因为部分资产挂多个特殊 theme）
- 但比保留 museum_collection 时接近很多
- "文化主题" 标题旁的"一件可挂多个"小字提示继续存在，用于向用户解释

## 2026-05-13（下午 9）｜THEME_DICT 系统性扩词典（33% → 100%）

### 背景
PM 发现"文化主题"大量资产落入 others 兜底，质疑打标率过低。抽样 340 件 NPM 数据审计后，初版词典**命中率仅 33.5%**——主要原因是原词典偏重"题材"而忽略了博物馆藏品的主体："青铜器、瓷器、玉器、漆器、珐琅、金银器"这些工艺品类**完全没词**。

### 分析路径
1. **写脚本扫描** 本地 NPM 340 件真实字段（category/tagZh/medium/title 二元组）
2. 发现高频类目：銅器/金屬器 89 件 + 木竹漆器 64 件 + 琺瑯器 52 件 + 玉石器 48 件 —— 这四类占 75%，原词典全部漏网
3. 高频媒介词：青銅/漆/玉石/紙/墨/瑪瑙/象牙/犀角 —— 同样一个不在
4. 高频纹饰/器物词：琺瑯/剔紅/銅胎/掐絲/雕漆/龍紋/雲龍/獸面紋/鼻煙壺/蒔繪 —— 都是博物馆核心词汇

### 决策：craft_heritage 吸收"物质文明/工艺"大类
讨论过是否新开 `material_culture`（物质文明）theme，但：
- 产品定位是 IP 改编向，分类要克制
- "青铜鼎/漆器/景泰蓝"本质都是"工艺遗产"，craft_heritage 英文名本身可理解为"工艺遗产"
- 按 8 theme 不变原则，扩词典即可

### 扩词典规模（每个 theme 中英双语系统性扩充）

|---|---|---|---|

### 两个关键"反误杀"修复
扩词典后跑审计发现两个显著误杀，针对性修复：

**1. 埃及神 "ra" → "ra-horakhty"**
单词 "ra" 作为 includes 子串会误命中 "dragon/era/doctor" 等 38 次。改为复合词"ra-horakhty"只匹配真正的"拉·霍拉克提"语境。

**2. 道教单字"道" 删除**
"道"字单独匹配会命中"道统/知道"等日常用词 29 次。删除保留"道教/太上/真人/八仙"等复合词。

**3. 帝王年号歧义修复（重要设计决策）**
初版把"乾隆/康熙/雍正/秦始皇/唐太宗/武则天/慈禧"都加入 historical_figure。审计发现其中"乾隆"85 次命中里 **48 次是年号用法**（"乾隆時收集成盒"/"乾隆年製"），只有 6 次是人物指代（"乾隆御製"）。

**解决策略**：从词典中**移除裸年号**，保留纯人物型名字（孔子/老子/关羽/王羲之/芭蕉等）。理由：
- 博物馆上下文里帝王年号 90%+ 是时期标记，不是人物指代
- 6 次真正人物命中不值得引入正则上下文匹配机制让词典变复杂
- 要查"乾隆御製"可走全文搜索
