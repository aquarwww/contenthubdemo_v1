## 三、内容力专区分类体系建议

### 3.1 建议采用"双轴分类"体系

```
维度一（纵轴）：文化圈 / 地域
维度二（横轴）：素材类型 / 用途
```

这样做的好处是：游戏项目组可以**按文化圈找灵感**（如"我想做一个宋代背景的关卡"），也可以**按素材类型找资源**（如"我需要纹样贴图参考"）。

### 3.2 文化圈分类（建议一级导航）

| 分类名 | 覆盖内容 | 主力数据源 |
|--------|---------|-----------|
| 🇨🇳 **华夏文明** | 中国历代（商周→明清） | 台北故宫、Met亚洲部、数字敦煌、故宫 |
| 🇯🇵 **东瀛物语** | 日本（飞鸟→明治） | Japan Search、ToMuCo、ColBase |
| 🇰🇷 **朝韩风华** | 朝鲜半岛（高句丽→朝鲜） | 韩国e-Museum、Met韩国藏品 |
| 🏛️ **古希腊罗马** | 地中海古典文明 | Met、Cleveland、Getty |
| 🏺 **古埃及** | 尼罗河文明 | Met、Walters、Cleveland |
| ☪️ **伊斯兰世界** | 中东/北非/中亚/安达卢斯 | Met、Walters、V&A |
| 🏰 **中世纪欧洲** | 5-15世纪欧洲 | Met、Walters、Europeana |
| 🎨 **文艺复兴至近代** | 欧洲15-19世纪 | AIC、Rijksmuseum、NGA、Getty |
| 🌏 **丝绸之路** | 跨文明交汇带 | Met、Smithsonian、Japan Search |
| 🎭 **设计与工艺** | 纹样/纺织/陶瓷/金属工艺 | Cooper Hewitt、V&A、Rijksmuseum |

### 3.3 素材类型分类（建议二级标签）

| 类型标签 | 说明 | 游戏用途 |
|---------|------|---------|
| 🖼️ **绘画与书画** | 油画、水墨画、壁画、版画 | 场景概念参考、UI 背景 |

| ✨ **纹样与图案** | 几何纹、植物纹、动物纹、藻井 | 纹理贴图、UI 装饰元素 |

## 四、各数据源详细信息

### 🇺🇸 欧美数据源

#### 1. The Metropolitan Museum of Art ✅已接入
- **Base URL**: `https://collectionapi.metmuseum.org/public/collection/v1`
- **认证**: 无需 Key | **速率**: 80次/秒 | **授权**: CC0
- **端点**: `/search?q=keyword` → `/objects/{id}`
- **规模**: 47万+藏品, 49万+图像

#### 2. Art Institute of Chicago
- **Base URL**: `https://api.artic.edu/api/v1`
- **认证**: 无需 Key | **授权**: CC0
- **端点**: `/artworks?q=keyword` → `/artworks/{id}`
- **图片**: IIIF `https://www.artic.edu/iiif/2/{image_id}/full/843,/0/default.jpg`
- **规模**: 12万+

#### 3. Cleveland Museum of Art
- **Base URL**: `https://openaccess-api.clevelandart.org/api/artworks/`
- **认证**: 无需 Key | **授权**: CC0
- **端点**: `?q=keyword` (搜索直接返回图片URL)
- **规模**: 6.4万+, 3.7万+图

#### 4. Rijksmuseum

## 五、技术接入要点速查

### 5.1 统一数据适配层设计

所有数据源通过 Adapter 模式归一化为统一的 `ContentItem` 模型：

```
┌───────────────────────────────────┐
│         前端 UI（统一体验）         │
└────────────────┬──────────────────┘
                 ▼
┌───────────────────────────────────┐
│        统一数据适配层 (BFF)         │
│   MetAdapter  │  AICAdapter  │…   │
└──┬────────┬───┴──────┬────────┬──┘
   ▼        ▼          ▼        ▼
  Met    Chicago    JapanSearch  ...
```

### 5.2 各 API 接入复杂度速查

| 类型 | 数据源 | 接入方式 |
|------|--------|---------|
| **直接 REST** | Met, AIC, Cleveland, Walters, V&A, ToMuCo | 前端直连或简单代理 |
| **REST + Key** | Rijksmuseum, Smithsonian, Harvard, Cooper Hewitt, Europeana | 后端代理（隐藏Key） |
| **IIIF 图像** | AIC, Rijksmuseum, Getty, ColBase, ToMuCo, 国会图书馆 | IIIF URL 拼接 |
| **SPARQL** | Japan Search | 需要编写 SPARQL 查询 |
| **批量下载** | 台北故宫, Cleveland(GitHub) | 定期同步到本地 |
| **人工获取** | 故宫, 敦煌, 国博, 古籍 | 人工下载+元数据录入 |

### 5.3 跨数据源搜索策略

```
用户输入中文关键词（如"宋代瓷器"）
    ↓
翻译层：生成多语言搜索词
  → 英文: "Song dynasty ceramics"
  → 日文: "宋時代 陶磁器"
    ↓
并行请求多个数据源 Adapter
    ↓
各 Adapter 归一化为 ContentItem[]
    ↓
合并 → 去重 → 按相关性排序 → 标注数据来源和授权
    ↓
前端统一渲染
```

### 6.3 版权合规体系建议

这是**最重要但最容易被忽视的环节**。建议从一开始就建立：

| 层级 | 做什么 | 怎么做 |
|------|--------|--------|
| **素材级** | 每条素材标注授权类型 | CC0 / CC BY / 需授权 → 用颜色标签区分 |
| **下载级** | 下载时弹出授权须知 | CC0 → 直接下载；CC BY → 显示署名要求；需授权 → 跳转申请 |
| **项目级** | 项目组使用前确认合规 | 提供「素材合规自检清单」模板 |
| **平台级** | 定期审核授权状态 | 关注数据源授权政策变更 |
